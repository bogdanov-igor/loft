# tz-write — написание ТЗ

Используй, когда владелец просит написать ТЗ, раздел, доработку или
изменение к существующему документу. Для чужой структуры сначала
`tz-adapt`; для нового проекта без структуры — `spec-bootstrap`.

## Порядок

1. **Структура.** Возьми профиль: `spec/_STRUCTURE.md` проекта или
   профиль из `memory/structures/`. Нет ни того ни другого →
   `spec-bootstrap` (новый проект) или `tz-adapt` (чужой документ).
2. **Тип и полнота запроса.** Определи тип задачи (интеграция,
   миграция, UI/UX, функциональность, баг, общее). Запрос неполный →
   `tz-elicit`: спроси пачкой, критичное без ответа → `QUESTIONS.md`.
3. **Источники.** Собери базу scout'ами: релевантные страницы
   `wiki/`, соседние ТЗ в `spec/`, глоссарий, memory. Большие пачки
   страниц — параллельные scout'ы, не главное окно. Зафиксируй список
   источников — он пойдёт в шапку документа.
4. **Черновик.** Пиши по профилю структуры. Дисциплина утверждений:
   - факт из корпуса → `[[ссылка]]` на страницу-источник;
   - внешний факт (стандарт, API) → ссылка на источник;
   - решение владельца → так и напиши: «решение от <дата>»;
   - остальное → пометка **(Допущение)** с обоснованием или вопрос
     в `QUESTIONS.md` и пометка **(Открытый вопрос → Q-N)**.
   Таблицы — GFM; диаграммы — ```mermaid; модели данных — ```dbml.
   Frontmatter: `status: draft`, `source:` (запрос), `based_on:` (источники).
5. **Самосверка.** Пройди по запросу пункт за пунктом: всё ли покрыто,
   нет ли отсебятины. Прогони `link-check` по затронутым файлам.
6. **Приёмка.** Агент `verifier`: утверждения + пути к запросу и
   источникам. fail → исправь и повтори. Вердикт pass → `status: review`
   (финальный `approved` ставит владелец).

## Правила

- Не выдумывай: нет источника и нет ответа — это допущение или вопрос,
  и оно видно в тексте. Скрытое предположение в ТЗ — дефект.
- Терминология — из глоссария проекта; нового термина нет в глоссарии →
  добавь туда, не плоди синонимы.
- Полное ТЗ или переработка нескольких документов — это `stage`
  (brief до, report после); одиночный раздел — нет.
- Изменение существующего ТЗ: сначала прочитай документ целиком,
  правь минимальным диффом, не переформатируй чужие разделы.
