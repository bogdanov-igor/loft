# Шаблоны формулировок требований

banks.md отвечает «что спросить», этот файл — «как записать ответ
проверяемо». Требование, у которого нет способа провалиться на
проверке, — не требование, а пожелание.

## Выбор шаблона

| Ситуация | Шаблон |
|---|---|
| Поведение действует всегда | EARS: всеобщее |
| Реакция на событие | EARS: событийное |
| Поведение в течение состояния | EARS: состояние |
| Обработка ошибки/сбоя | EARS: нежелательное |
| Функция с опции/фичи | EARS: опциональное |
| Нужна привязка к ценности для роли | User story + критерии |
| Производительность, надёжность, объёмы | НФТ-таблица |

Каждое требование несёт: ID по конвенции проекта (если объявлена в
`spec/_STRUCTURE.md`), источник `[[wikilink]]` или пометку
**(Допущение)**, и критерий приёмки.

## EARS по-русски

- **Всеобщее**: «<Система> должна <функция>.»
  — *«Шлюз должен журналировать все входящие запросы.»*
- **Событийное**: «Когда <триггер>, <система> должна <реакция>
  [не позднее чем за <время>].»
  — *«Когда получен платёжный запрос, шлюз должен вернуть
  подтверждение не позднее чем за 3 с.»*
- **Состояние**: «Пока <состояние>, <система> должна <поведение>.»
  — *«Пока соединение с процессингом потеряно, шлюз должен ставить
  запросы в очередь повторной отправки.»*
- **Нежелательное**: «Если <нежелательное событие>, <система> должна
  <защитная реакция>.»
  — *«Если ответ процессинга не получен за 30 с, шлюз должен вернуть
  клиенту ошибку TIMEOUT и записать инцидент.»*
- **Опциональное**: «Там, где <опция включена>, <система> должна
  <поведение>.»
  — *«Там, где включён кэшбек, сервис должен начислять бонус при
  подтверждении транзакции.»*

Комбинируются: «Пока <состояние>, когда <триггер>, система должна…».
Запрещённые слова в требовании: «быстро», «удобно», «надёжно»,
«и т.д.», «по возможности» — вместо них число, перечень или отдельное
НФТ.

## User story + критерии приёмки

```markdown
Как <роль>, я хочу <действие>, чтобы <ценность>.

Критерии приёмки:
- Дано <контекст>, когда <действие>, тогда <проверяемый результат>.
- Дано <контекст ошибки>, когда <действие>, тогда <реакция и сообщение>.
```

Минимум два критерия: счастливый путь и хотя бы одна ошибка. Критерий
без «тогда <наблюдаемый результат>» — не критерий.

## НФТ-таблица

```markdown
| Параметр | Значение | Как измеряем | Источник |
|---|---|---|---|
| Время ответа, 95-й перцентиль | ≤ 500 мс | нагрузочный тест, профиль N | [[ссылка]] |
| Доступность | 99,9 % / месяц | мониторинг, окно обслуживания исключено | [[ссылка]] |
| Объём | 50 тыс. транзакций/сутки, пик ×3 | статистика за квартал | [[ссылка]] |
```

НФТ без колонки «Как измеряем» — декларация, не требование.

## Открытые концы

Не хватает данных для шаблона → это вопрос (`tz-elicit` → пачкой
владельцу, критичное в `QUESTIONS.md`), а не творческая формулировка.
Полузаполненный шаблон с пометкой **(Допущение: …)** лучше гладкого
абзаца без опоры.
