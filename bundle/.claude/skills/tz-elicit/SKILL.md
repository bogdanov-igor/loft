# tz-elicit — вопросы к запросу

Используй, когда запрос на ТЗ неполный (а он почти всегда неполный):
прежде чем писать, собери недостающее. Не по одному вопросу — пачкой.

## Порядок

1. Определи тип задачи и возьми соответствующий банк из
   `banks.md` (рядом с этим скиллом): интеграция / миграция / UI-UX /
   функциональность / баг / общее. Смешанный тип — объедини банки.
2. Вычеркни то, на что запрос и корпус уже отвечают (scout по wiki и
   spec: возможно, половина ответов уже написана).
3. Оставшиеся раздели на **критичные** (без ответа ТЗ будет неверным)
   и **уточняющие** (без ответа будет допущение).
4. Задай владельцу одним сообщением, сгруппировав по темам. По
   возможности предлагай варианты ответа — выбрать проще, чем
   формулировать.
5. Без ответа в этой сессии: критичные → `QUESTIONS.md` (с планом
   продолжения), работа по ним паркуется; уточняющие → пиши с
   пометкой **(Допущение: ...)** прямо в тексте.
6. Результат зафиксируй: короткий бриф (цель, скоуп, не-скоуп,
   ограничения, критерии приёмки) — в начало будущего ТЗ или в
   `stages/NNN/brief.md`, если работа большая.

## Правила

- Вопрос, ответ на который есть в корпусе, — не задаётся: сначала
  scout, потом владелец.
- Ответы владельца — источник: фиксируй их в документе как «решение
  от <дата>», а не пересказом без следа.
- Банки — стартовая точка, не потолок: специфичные для проекта вопросы
  добавляй по месту; повторяющиеся — дописывай в банк проекта
  (`spec/_STRUCTURE.md`, секция «Вопросы») и предложи внести в ядро.
