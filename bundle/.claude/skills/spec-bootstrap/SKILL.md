# spec-bootstrap — новый проект

Используй один раз на проект: когда корпуса ещё нет или он — россыпь
файлов без структуры. Это работа уровня `stage`, если проект не пустой
(есть что инвентаризировать).

## Порядок

1. **Инвентаризация** (если файлы уже есть): scout по корню — что за
   файлы, какие форматы, что из этого источники (в `inbox/`), что
   черновики (в `spec/`), что мусор (кандидаты на удаление — списком
   владельцу, не удалять молча).
2. **Раскладка.** Создай недостающее из стандартной раскладки
   контракта: `spec/`, `inbox/`, `wiki/` — только если есть Confluence.
3. **Профиль структуры — `spec/_STRUCTURE.md`.** Согласуй с владельцем
   ДО написания документов:
   - дерево разделов типового ТЗ проекта (обязательные/опциональные);
   - стиль: заголовки, нумерация требований, оформление таблиц и
     диаграмм, статусная модель (`draft → review → approved`);
   - соглашение об именах файлов (латиница-слаги или кириллица — одно
     из двух, зафиксировать);
   - секция «Вопросы»: проектные добавки к банкам `tz-elicit`.
4. **Опорные документы.** `spec/00-обзор.md` (что за проект, системы,
   роли, ссылки), `spec/глоссарий.md` (термин — определение — источник),
   `spec/решения.md` (реестр решений: дата, решение, кто, обоснование).
5. **Связывание.** Все созданные страницы связаны `[[wikilink]]`-ами
   от `spec/00-обзор.md`; прогони `link-check`.
6. **Фиксация.** Профиль структуры — однострочник в `memory/MEMORY.md`;
   если проект чужой (структура заказчика) — профиль живёт в
   `memory/structures/` (skill `tz-adapt`).

## Правила

- Не порождай пустых заглушек «на будущее»: раздел появляется, когда
  есть что писать. Пустая структура — это `_STRUCTURE.md`, а не десять
  пустых файлов.
- git: если проект не под git — предложи владельцу `git init` первым
  пунктом (история правок ТЗ = машина времени бесплатно). Настаивать —
  нет, предложить — обязательно.
- Кириллица в именах папок/файлов допустима, но реши это в п.3 явно:
  смесь кириллицы и транслита в одном корпусе — дефект.
