# stage — протокол большой работы

Используй, когда работа охватывает много документов, несёт риск
(перезапись корпуса, массовые переименования, сдача заказчику) или
не влезает в одну сессию. Иначе — не используй: это правило двух
уровней.

## Файлы

`stages/NNN-slug/` (следующий свободный NNN, три цифры):

**`brief.md`** — до начала:

- Цель: одно проверяемое предложение.
- Опора: релевантные заметки памяти `[[слаг]]`-ами, или честное
  «прецедентов нет» — пустая опора лучше декоративной цитаты.
- Подход + 1–2 отвергнутые альтернативы (по строке: почему проиграли).
- Юниты: нумерованный список. На юнит: ожидаемый результат
  (формулировка-проверка, которая может провалиться, а не пожелание)
  и откат (команда, «git revert» или «восстановить из inbox/»).

**`report.md`** — при закрытии:

- По юнитам: что сделано (пути, документы), свидетельства ожидаемого
  результата, вердикт verifier.
- Итог: сделано / частично / брошено — с честной причиной.

Два файла. Без plan.json, без sha256, без локов, без run id.

## Правила

- Спорный скоуп → покажи владельцу `brief.md` до исполнения. Молчание
  не гноит работу: правило парковки контракта — вопрос в
  `QUESTIONS.md`, работа над другими юнитами продолжается.
- Параллельные юниты — только по непересекающимся файлам (один
  документ — один юнит); параллельные правки одних файлов теряют
  работу.
- Вердикт отчёта выносит агент `verifier` со свежим контекстом:
  вставь его сводную строку и ссылки на свидетельства. Отчёт без
  verifier-прохода = `частично`, что бы ни заявляли юниты.
- Незакрытые юниты при закрытии stage → строки в `BACKLOG.md`;
  молчаливых потерь не бывает.
- Stage чему-то научил → skill `remember` до закрытия.
