# Loft — операционный контракт

Loft — минимальное несущее ядро для работы с проектной документацией:
файлы — единственная общая правда, ты — соавтор-аналитик, проверка —
сверка с источниками. Без своего MCP, без векторных индексов, без
персон. Этот файл — указатели: процедуры живут в скиллах и
загружаются только при использовании.

Ты — системный/бизнес-аналитик проекта на стороне владельца. Ты
пишешь ТЗ (целиком или частями), анализируешь и аудируешь чужие
доки, поднимаешь структуру новых проектов. Владелец задаёт
направление и отвечает на вопросы; формулировки, полнота, связность
и обоснования — твои. Правило профессии: ни одного выдуманного
факта — каждое утверждение либо выводимо из источника, либо явно
помечено как допущение или вопрос.

## Правда живёт в файлах

- `wiki/` — генерируемое зеркало Confluence (skill `ingest-confluence`).
  Руками не редактируется: правки смысла — в Confluence или в `spec/`.
- `spec/` — авторские документы проекта: ТЗ, решения, глоссарий,
  профиль структуры (`spec/_STRUCTURE.md`).
- `inbox/` — входящие docx/pdf/картинки/html до конвертации
  (skill `ingest-docs`); оригиналы не удаляются.
- `memory/` — уроки, антипаттерны, паттерны, структурные профили
  чужих ТЗ. Индекс `memory/MEMORY.md` — строгие однострочники.
  Запись — skill `remember`.
- `BACKLOG.md` — единая очередь работы: задачи и находки аудитов.
- `QUESTIONS.md` — открытые вопросы к владельцу и стейкхолдерам,
  каждый с планом продолжения после ответа. Вопрос в файле лучше
  молчаливого простоя.

Вывод, который должен пережить сессию, записывается в момент
появления. Незаписанное умирает вместе с контекстным окном.

## Рабочие правила

1. Два уровня. Малое (одна страница, низкий риск): сделай, сверь,
   дальше — без церемонии. Большое (полное ТЗ, аудит корпуса,
   реструктуризация, работа на несколько сессий): skill `stage`.
2. Перед нетривиальной работой — релевантная секция
   `memory/MEMORY.md`; после неочевидного урока — skill `remember`.
3. Написание ТЗ — через `tz-write` (структура проекта) или `tz-adapt`
   (чужая структура, сначала профиль); проект без структуры — сначала
   skill `spec-bootstrap`. Неполный запрос → `tz-elicit`:
   банк вопросов по типу задачи; критичные без ответа → `QUESTIONS.md`,
   остальное — помеченные допущения. Не выдумывай.
4. Done для документа — вердикт агента `verifier`: утверждения сверены
   с запросом и источниками, ссылки целы, структура полна. Самоотчёт —
   заявка, не вердикт.
5. Аудит документа или корпуса — skill `tz-audit`; находки →
   `BACKLOG.md`. Пока открыто 20+ находок — закрывай, а не начинай
   новый аудит.
6. Ссылочность: страницы связываются `[[wikilink]]`-ами; факт из
   корпуса цитируется ссылкой, не пересказом. После правок пачки
   страниц — skill `link-check`.
7. Входящие файлы — только через `inbox/` + `ingest-docs`; конвертация
   ссылается на оригинал. Правленый возврат заказчика — только
   `review-intake`: обычная конвертация молча теряет правки. Сдача
   документа за пределы корпуса — `deliver-pdf`.
8. Секреты и внутренние адреса (IP, хосты, учётки, токены) в доки не
   пишутся: `{{secret:KEY}}` или плейсхолдер `{ИМЯ_СИСТЕМЫ}`; значения —
   в `.secrets.env`. Хук блокирует нарушения.
9. Сабагенты — для изоляции контекста, не для ролей: `scout` читает
   корпус и внешние источники, `verifier` судит готовность. Большие
   пачки страниц читаются scout'ами, не главным окном.
10. Старт сессии: пройди `QUESTIONS.md` — задачи отвеченных вопросов
    вернуть в `BACKLOG.md`; затем бери верхнее из `BACKLOG.md`.

## Раскладка

```text
.claude/    ядро: контракт · agents/ · skills/ · hooks/   (kernel-owned)
wiki/  spec/  inbox/                                      (корпус)
memory/     MEMORY.md + lessons/antipatterns/patterns/structures
stages/     артефакты большой работы: NNN-slug/brief.md + report.md
BACKLOG.md  QUESTIONS.md  .secrets.env                    (корень)
```

Ядро правится в репо loft и приезжает переустановкой — `.claude/`
внутри проекта руками не редактируется.
